Chapter 1

The Storage Engine & Disk Anatomy

Understanding how data actually lives on the metal.

Max keys per page: 3. Watch the page sort your keys, and then split/promote the median when it overflows.

When you map an object in Spring Data JPA using @Entity, the framework abstracts away the brutal reality of the hardware. It makes you feel like data is just a floating collection of Java objects. But beneath the ORM, beneath the SQL parser, and beneath the query optimizer, lies the Storage Engine. This is where software hits the metal.

1. The Illusion of the Table

A "table" does not exist on a hard drive. The hard drive only understands magnetic polarity (HDD) or trapped electrons (SSD) arranged in massive, linear arrays of blocks. The database storage engine's entire job is to take your beautifully structured 2D table relationships and serialize them into a 1D stream of bytes that can be written to, and read from, physical disk sectors.

2. The Fundamental Unit of I/O: The Page

Databases do not read or write individual rows. If you run SELECT name FROM users WHERE id = 5;, the database does not pluck exactly 15 bytes off the disk. It reads a Page (or Block).

A Page is a fixed-size chunk of memory, typically 8KB (PostgreSQL) or 16KB (InnoDB/MySQL). When you request a single row, the disk controller fetches the entire 16KB page that contains your row, along with dozens of other rows sitting adjacent to it, and loads that entire page into RAM.

Senior Architect Insight: This is the root cause of 90% of backend performance bottlenecks. If your query requires scanning 10,000 unindexed rows spread across 500 different pages, your system will suffer massive I/O latency. Disk I/O is orders of magnitude slower than CPU cache or RAM operations.

3. Extents and Segments

To prevent file fragmentation, the storage engine doesn't just ask the operating system for one page at a time. It allocates memory in Extents. An extent is usually a collection of contiguous pages (e.g., 64 pages of 16KB = 1MB extent). Grouping extents together forms a Segment, which correlates to a specific table or index.

By keeping data physically contiguous on the disk, the storage engine optimizes for Sequential I/O, which is incredibly fast even on older spinning hard drives, because the read head doesn't have to physically move to different tracks.

4. The Buffer Pool: Hiding the Latency

Because disk reads are slow, every major DBMS uses a Buffer Pool (RAM). When a page is read from the disk, it is cached in the Buffer Pool. The next time you request that data, it is served directly from RAM (a logical read) rather than the disk (a physical read).

When you update an entity in Spring Boot, you aren't writing to the disk immediately. You are updating the page in the Buffer Pool, marking it as a "Dirty Page". A background thread (like InnoDB's page cleaner) will asynchronously flush these dirty pages back to the physical disk later to ensure durability.

5. The B-Tree: Taming the Chaos

If data is just written to random pages as it arrives, finding a specific record requires a Full Table Scan—reading every single page into the Buffer Pool until the record is found. To solve this, we use the B-Tree (Balanced Tree) Index.

A B-Tree structures data in a hierarchical tree where each node maps to a physical database page. Instead of scanning millions of rows, the database traverses the tree, jumping from the Root Page to Branch Pages, and finally to the Leaf Page containing the exact physical address of your data. This reduces search time from $O(N)$ to $O(\log N)$.
