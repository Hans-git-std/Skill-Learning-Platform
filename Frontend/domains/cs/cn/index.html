Chapter 1

The OSI Model & Packet Encapsulation

How data travels from your application to the physical wire.

When you type a URL into your browser and hit Enter, it feels instantaneous. But beneath that single click, your operating system orchestrates one of the most complex handoffs in all of computing. Your HTTP request must be wrapped, re-wrapped, and wrapped again—each layer adding its own header—before a single electrical pulse ever leaves your network card.

1. The Layered Abstraction

The OSI (Open Systems Interconnection) Model divides network communication into seven distinct layers. Each layer talks only to the layer directly above and below it. This separation of concerns is the reason the internet scales to billions of devices: you can swap out Wi-Fi for Ethernet at Layer 2 without rewriting your web browser at Layer 7.

In practice, most of the internet runs on the simpler TCP/IP model (4 layers), but understanding OSI gives you the vocabulary to debug any network problem—from a misconfigured DNS server to a dropped TCP packet.

2. Encapsulation: Wrapping Data at Each Layer

When your application sends data, it passes downward through the stack. At each layer, a header (and sometimes a trailer) is prepended to the payload from the layer above. This process is called Encapsulation.

  • Layer 7 (Application): Your HTTP request—plain text like GET /index.html HTTP/1.1.
  • Layer 4 (Transport): TCP adds source/destination ports, sequence numbers, and checksums for reliable delivery.
  • Layer 3 (Network): IP adds source/destination IP addresses so routers know where to forward the packet.
  • Layer 2 (Data Link): Ethernet adds MAC addresses for local delivery on the same subnet.
  • Layer 1 (Physical): The frame is converted to electrical signals, light pulses, or radio waves.
Network Engineer's Reality Check: When you run tcpdump or Wireshark, you are literally peeling back these layers in reverse (decapsulation). A "TCP retransmission" error means Layer 4 detected a lost segment. An "TTL exceeded" error means Layer 3's Time-To-Live counter hit zero. Knowing which layer failed tells you exactly where to look.

3. MTU and Fragmentation

Every link in the network has a Maximum Transmission Unit (MTU)—typically 1500 bytes for Ethernet. If your encapsulated packet exceeds the MTU, Layer 3 (IP) must fragment it into smaller pieces. Fragmentation is expensive and is a common cause of subtle performance bugs. This is why Path MTU Discovery exists: to find the smallest MTU along the entire route before sending.

Chapter 2

IP Addressing & Packet Routing

How routers decide where your packet goes next.

Every device on the internet has an IP address—a 32-bit number (IPv4) or 128-bit number (IPv6) that uniquely identifies it on the network. But IP addresses alone don't tell a packet how to reach its destination across continents. That job belongs to routers and their routing tables.

1. Subnetting and CIDR Notation

IP addresses are grouped into subnets using CIDR notation like 192.168.1.0/24. The /24 means the first 24 bits identify the network, and the remaining 8 bits identify individual hosts (256 addresses, 254 usable). Your home router typically owns 192.168.1.0/24, assigning addresses like 192.168.1.5 to your laptop via DHCP.

2. Longest Prefix Match

When a router receives a packet destined for 93.184.216.34, it doesn't "know" the full path. It consults its routing table and finds the entry with the longest matching network prefix. If the table contains both 93.184.0.0/16 and 0.0.0.0/0 (the default route), the /16 entry wins because it is more specific.

Architect's Insight: This is why a misconfigured default route can black-hole all traffic, while a missing specific route only breaks one subnet. Production networks use BGP (Border Gateway Protocol) to exchange routing tables between autonomous systems—the protocol that literally holds the internet together.

3. NAT: Stretching IPv4

Your laptop's IP 192.168.1.5 is a private address—unroutable on the public internet. When your packet leaves your home, your router performs Network Address Translation (NAT), rewriting the source IP to your public IP and tracking the connection in a translation table. NAT is why millions of devices can share a single public IP address.

4. TTL and the Traceroute Tool

Every IP packet carries a Time-To-Live (TTL) counter. Each router decrements it by 1. When TTL hits zero, the router drops the packet and sends back an ICMP "Time Exceeded" message. The traceroute command exploits this: it sends packets with TTL=1, then TTL=2, then TTL=3, mapping every hop along the path to a destination.

Chapter 3

TCP & The Three-Way Handshake

Building reliable connections on an unreliable network.

IP (Layer 3) is a best-effort protocol—it delivers packets with no guarantee of order, delivery, or integrity. TCP (Layer 4) sits on top of IP and provides reliable, ordered, byte-stream delivery. Before any HTTP request can be sent, TCP must first establish a connection using the famous Three-Way Handshake.

1. Why a Handshake?

Both sides must agree on initial sequence numbers, confirm they can receive data, and synchronize their state machines. Without this, a client could send data to a server that isn't listening, or two stale packets from a previous connection could corrupt a new session.

2. The Three Steps

  • Step 1 — SYN: Client sends a TCP segment with the SYN flag set and a random sequence number (e.g., Seq=100). Client enters SYN-SENT state.
  • Step 2 — SYN-ACK: Server responds with SYN+ACK flags, its own sequence number (Seq=300), and Ack=101 (client's Seq + 1). Server enters SYN-RCVD state.
  • Step 3 — ACK: Client sends ACK with Ack=301. Both sides enter ESTABLISHED state. Data transfer can begin.
Performance Note: This handshake adds one full round-trip time (RTT) before any data flows. On a mobile network with 200ms RTT, that's 200ms of pure latency before your API request even starts. This is why HTTP/3 moved to QUIC (which uses UDP and combines the handshake with TLS negotiation into 0-RTT or 1-RTT).

3. Flow Control and Congestion Control

Once connected, TCP manages two critical mechanisms. Flow Control (via the receive window) prevents a fast sender from overwhelming a slow receiver's buffer. Congestion Control (Slow Start, Congestion Avoidance) prevents a sender from overwhelming the network itself. When packet loss is detected (via duplicate ACKs or timeout), TCP reduces its sending rate—this is why your download speed fluctuates during network congestion.

4. Connection Teardown

Closing a TCP connection requires a Four-Way Handshake (FIN → ACK → FIN → ACK) because each side must independently signal it has no more data to send. A common bug in backend development is leaving connections in TIME-WAIT state, exhausting available ports on high-traffic servers.

Chapter 4

DNS: The Internet's Phone Book

Translating human-readable names into IP addresses.

Humans remember example.com. Routers understand 93.184.216.34. The Domain Name System (DNS) is the distributed database that bridges this gap. Every single web request, email, and API call begins with a DNS lookup—and a slow DNS response adds latency to every downstream operation.

1. The DNS Hierarchy

DNS is organized as a tree. At the root are the Root Servers (13 logical clusters, lettered A through M). Below them are TLD (Top-Level Domain) servers for .com, .org, .in, etc. Each domain has an Authoritative Nameserver that holds the actual records.

2. Recursive vs. Iterative Queries

When you query example.com, your browser doesn't talk to root servers directly. It asks your ISP's Recursive Resolver (often at 8.8.8.8 or 1.1.1.1). The resolver does the hard work: it queries root → TLD → authoritative server, caches the result, and returns the final IP to you.

Production Insight: DNS caching is controlled by the TTL (Time-To-Live) field in each record. A TTL of 300 seconds means resolvers cache the answer for 5 minutes. During a migration, engineers lower TTL days in advance so DNS changes propagate quickly. A TTL of 86400 (24 hours) means a bad deploy can take a full day to roll back via DNS alone.

3. Record Types That Matter

  • A Record: Maps a hostname to an IPv4 address.
  • AAAA Record: Maps to IPv6.
  • CNAME: Alias—points one name to another (e.g., www → example.com).
  • MX Record: Mail exchange—tells email servers where to deliver mail.
  • NS Record: Delegates a domain to its authoritative nameservers.

4. DNS Security: DNSSEC and DoH

Classic DNS is unencrypted and unauthenticated—making it vulnerable to spoofing and man-in-the-middle attacks. DNSSEC adds cryptographic signatures to verify records haven't been tampered with. DNS over HTTPS (DoH) encrypts DNS queries inside TLS, preventing ISPs from snooping on your browsing history.
