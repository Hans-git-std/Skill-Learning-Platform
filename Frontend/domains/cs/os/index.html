Chapter 1

Process Lifecycle & The Process Control Block

How the kernel tracks every running program on your machine.

When you double-click an application, it feels like the program simply "starts." But the operating system does not launch programs—it creates processes. A process is not the executable file on disk; it is a living, breathing instance of that program loaded into RAM, with its own memory space, file descriptors, and a unique identity tracked by the kernel.

1. Process vs. Thread

A process is an isolated container: its own virtual address space, heap, stack, and system resources. A thread is a lightweight unit of execution inside a process. All threads in a process share the same memory, but each has its own program counter and stack. When Chrome opens 40 tabs, you have one browser process with dozens of threads—or in modern Chrome, multiple processes for sandboxing.

2. The Process Control Block (PCB)

The kernel cannot manage thousands of processes by guessing. For every process, it maintains a Process Control Block (PCB)—a data structure in kernel memory containing:

  • Process ID (PID): Unique identifier (e.g., PID 4821).
  • Process State: New, Ready, Running, Waiting, or Terminated.
  • Program Counter: The exact instruction address to resume from after a context switch.
  • CPU Registers: Saved snapshot of register values when the process was preempted.
  • Memory Limits: Page table pointer, base/limit registers for virtual memory.
  • I/O Status: List of open files, network sockets, and pending I/O requests.
Kernel Engineer's Reality Check: When you run ps aux or open Task Manager, you are viewing a filtered export of PCB data. When a process crashes with a segfault, the kernel reads the saved program counter from the PCB to tell you exactly which instruction failed.

3. The Five Process States

Every process moves through a finite state machine during its lifetime:

  • New: Process is being created. OS allocates PCB and memory.
  • Ready: Process is loaded and waiting for CPU time. Sits in the Ready Queue.
  • Running: Process has the CPU. Only one process per core can be Running.
  • Waiting (Blocked): Process is waiting for I/O, a lock, or a signal. Removed from Ready Queue.
  • Terminated: Process finished or was killed. PCB is cleaned up, resources reclaimed.

4. Context Switching: The Hidden Tax

When the scheduler preempts Process A to run Process B, the CPU does not magically jump. The kernel performs a Context Switch: it saves Process A's entire register state into its PCB, loads Process B's registers from its PCB, and jumps to B's program counter. This takes microseconds—but on a server handling 10,000 context switches per second, that overhead becomes measurable. This is why excessive threading without purpose hurts performance.

Chapter 2

CPU Scheduling Algorithms

Deciding which process gets the processor when everyone wants it.

On a single-core machine, only one process can execute at any instant. Yet you have hundreds of processes in the Ready Queue—all waiting for their turn. The CPU Scheduler is the kernel component that picks the next process to run. The algorithm it uses determines your system's responsiveness, throughput, and fairness.

1. Scheduling Criteria

There is no perfect scheduler. Every algorithm optimizes for different goals:

  • CPU Utilization: Keep the CPU busy. Idle CPU is wasted silicon.
  • Throughput: Maximize processes completed per unit time.
  • Turnaround Time: Minimize total time from submission to completion.
  • Waiting Time: Minimize time spent in the Ready Queue.
  • Response Time: Minimize time until a process first gets the CPU (critical for interactive apps).

2. First-Come, First-Served (FCFS)

The simplest scheduler: processes run in arrival order, non-preemptively. If Process A needs 24ms and arrives first, Processes B and C wait the entire 24ms—even if they only need 3ms each. This is the Convoy Effect: a short job stuck behind a long job suffers massive waiting time. FCFS is fair in order but terrible for response time.

3. Round Robin (RR)

The workhorse of time-sharing systems (Linux, Windows, macOS). Each process gets a fixed time quantum (typically 4–100ms). If a process doesn't finish within its quantum, it is preempted and moved to the back of the Ready Queue. This guarantees no process waits more than (n-1) × quantum milliseconds, giving excellent response time for interactive workloads.

Production Insight: Linux's Completely Fair Scheduler (CFS) is a sophisticated evolution of Round Robin. Instead of fixed quanta, it assigns each process a "virtual runtime" and always picks the process that has consumed the least CPU time—ensuring proportional fairness across priorities.

4. Priority Scheduling & Starvation

Each process gets a priority number. The highest-priority Ready process runs next. Real-time systems (medical devices, flight control) use strict priority scheduling. The danger is Starvation: a low-priority process may never run if high-priority processes keep arriving. The fix is Aging—gradually increasing the priority of processes that have waited too long.

5. Multilevel Feedback Queue

Modern OS schedulers combine multiple queues with different priorities and time quanta. A new process starts in the highest-priority queue. If it uses its full quantum, it is demoted to a lower queue. I/O-bound processes (interactive apps) stay in high queues; CPU-bound processes (video encoding) sink to lower queues. This adapts to process behavior without manual configuration.

Chapter 3

Virtual Memory & Page Replacement

How your OS gives every process the illusion of infinite RAM.

Your laptop might have 16GB of physical RAM, but every process believes it owns 128TB of address space. This illusion is Virtual Memory—one of the most important abstractions in operating systems. It lets the OS run programs larger than physical RAM, isolate processes from each other, and share memory efficiently.

1. Paging: Fixed-Size Memory Blocks

Virtual memory divides address space into fixed-size pages (typically 4KB). Physical RAM is divided into matching frames. The Page Table maps each virtual page number to a physical frame number—or marks it as not present. When Process A accesses virtual address 0x00401000, the MMU (Memory Management Unit) consults the page table, translates it to a physical frame, and accesses RAM.

2. The Page Fault

If the page table entry says "not in memory," the CPU triggers a Page Fault—a hardware interrupt. The kernel's page fault handler:

  • Finds a free frame (or evicts one using a replacement algorithm).
  • Loads the page from disk (swap space) into the frame.
  • Updates the page table entry.
  • Restarts the instruction that caused the fault.

Page faults are expensive—disk I/O takes milliseconds vs. nanoseconds for RAM. Excessive page faults cause thrashing, where the system spends more time swapping than executing.

Systems Insight: When your IDE feels sluggish with 50 tabs open, you are likely thrashing. The OS is constantly evicting and reloading pages because working set size exceeds RAM. Adding RAM or closing apps reduces page faults dramatically.

3. The Translation Lookaside Buffer (TLB)

Page table lookups on every memory access would be too slow. The CPU caches recent translations in the TLB—a small, hardware-managed associative cache. A TLB hit takes ~1 cycle; a TLB miss requires a full page table walk (multiple memory accesses). Context switches flush the TLB, which is why switching between processes has a measurable cost.

4. Page Replacement Algorithms

When all frames are full and a page fault occurs, the OS must evict a page. The algorithm matters:

  • FIFO (First-In, First-Out): Evict the oldest page. Simple but ignores usage patterns.
  • LRU (Least Recently Used): Evict the page not accessed for the longest time. Excellent approximation of optimal behavior.
  • Clock (Second Chance): Practical LRU approximation used by Linux. Pages get a reference bit; the clock hand sweeps and evicts pages with bit=0.

Chapter 4

Deadlocks & Synchronization

When processes wait forever—and how the kernel prevents it.

Modern software is concurrent. Multiple threads read and write shared data simultaneously. Without coordination, you get race conditions—two threads incrementing a counter and losing an update. With too much coordination, you get something worse: deadlock, where processes wait for each other forever.

1. Synchronization Primitives

The OS provides tools to coordinate access to shared resources:

  • Mutex (Mutual Exclusion Lock): Only one thread can hold the lock. Others block until it is released. Used to protect critical sections.
  • Semaphore: A counter-based lock. A binary semaphore (0 or 1) acts like a mutex. A counting semaphore allows N threads (e.g., limit DB connection pool to 10).
  • Monitor: High-level construct combining a mutex with condition variables. Java's synchronized blocks are monitors.

2. The Four Necessary Conditions for Deadlock

Deadlock can only occur when ALL four conditions hold simultaneously (Coffman conditions):

  • Mutual Exclusion: At least one resource can be held by only one process at a time.
  • Hold and Wait: A process holds at least one resource while waiting for another.
  • No Preemption: Resources cannot be forcibly taken; they must be voluntarily released.
  • Circular Wait: A cycle exists: P1 waits for P2, P2 waits for P3, P3 waits for P1.
Backend Developer's Nightmare: Deadlocks in production databases often manifest as queries hanging forever. PostgreSQL detects deadlocks automatically and kills one transaction—but only after a timeout. In application code, always acquire locks in a consistent global order to break the circular wait condition.

3. Deadlock Handling Strategies

Operating systems and databases choose one of four approaches:

  • Prevention: Break one of the four Coffman conditions (e.g., require all resources to be requested at once).
  • Avoidance: Banker's Algorithm—only grant resources if the resulting state is "safe."
  • Detection & Recovery: Periodically check the Resource Allocation Graph for cycles; kill or rollback one process.
  • Ignore: The Ostrich Algorithm. Used by most general-purpose OSes for user-level deadlocks—you handle it in your code.

4. The Banker's Algorithm

Before granting a resource request, the OS simulates the allocation and checks if a safe sequence exists—a ordering of processes where each can eventually finish and release its resources. If no safe sequence exists, the request is denied even if resources are available. This prevents deadlock at the cost of reduced resource utilization.
