Chapter 1

Data Representation & Arithmetic

Bridging the physical and numerical worlds.

Computer Architecture fundamentally concerns itself with how physical voltage states map to logical and numerical concepts. At the lowest level, all computation is merely the orchestration of these electrical states. When you declare an integer or a float in a high-level language, the processor has no inherent understanding of these types. It merely observes patterns of high (1) and low (0) voltages.

1. Integer Representation and Two's Complement

To represent unsigned integers, we use straightforward positional binary notation. However, expressing negative numbers poses a significant challenge since hardware lacks a physical "minus" symbol. Early architectures experimented with Sign-and-Magnitude and One's Complement, but these approaches suffered from a "negative zero" anomaly and required complex ALU designs to handle addition and subtraction as separate operations.

The solution is Two's Complement. By inverting all bits and adding one, we establish a system where the Most Significant Bit (MSB) effectively acts as a sign bit, yet the underlying binary addition logic remains identical for both positive and negative numbers. This brilliant mathematical abstraction allows the CPU to perform subtraction using the exact same hardware circuit (the adder) as addition.

2. IEEE 754 Floating-Point Standard

Scientific computing requires a vast dynamic range—from sub-atomic distances to intergalactic measurements. Fixed-point arithmetic is insufficient. Thus, the IEEE 754 standard was universally adopted, defining a representation that mimics scientific notation in binary.

A floating-point number is partitioned into three fields: the Sign bit, the Biased Exponent, and the Fraction (Mantissa). The exponent uses a bias rather than two's complement to make floating-point comparison as fast as integer comparison in hardware. The mantissa employs an implicit leading "1" to squeeze an extra bit of precision out of the physical memory. Understanding these physical hardware compromises is crucial for avoiding insidious rounding errors in software.

Chapter 2

Digital Logic & Microoperations

The foundation of computation.

Before we can execute complex instructions, we must build the physical structures that compute logic and route data. These are the microoperations—the atomic, indivisible steps that occur during a single clock cycle.

1. The Arithmetic Logic Unit (ALU)

The ALU is the mathematical heart of the processor. It is entirely composed of combinational logic gates. Through multiplexers, a control signal dictates whether the ALU's internal adder, AND gates, OR gates, or XOR gates are routed to the final output. The ALU does not hold memory; it merely continuously processes whatever voltages are currently presented to its input buses.

2. Register Transfers

Computation without data movement is useless. The architecture connects registers to the ALU via buses. A microoperation often looks like R3 ← R1 + R2. In a single clock cycle, the Control Unit asserts the read enable on R1 and R2, routes them onto the internal buses feeding the ALU, signals the ALU to perform addition, and asserts the write enable on R3 to capture the output on the clock's rising edge.

Chapter 3

Instruction Set Architecture (ISA)

The software-hardware contract.

The Instruction Set Architecture (ISA) is the defining boundary between software and hardware. It is the language that the compiler writes and the silicon natively understands (e.g., x86, ARM, RISC-V). An ISA dictates the available registers, data types, memory addressing modes, and the binary format of the instructions.

1. Instruction Formats and Fields

A machine instruction is a rigid binary string. In a fixed-length ISA like MIPS or ARM, every instruction is exactly 32 bits. These 32 bits are sliced into fields. An R-type (Register) instruction might dedicate 6 bits to the opcode (the "verb" like ADD or SUB), 5 bits to the source register, 5 bits to the target register, and so on. The hardware's Decoder physically wires these specific bit-ranges to different parts of the CPU datapath.

2. RISC vs. CISC Philosophies

Historically, memory was incredibly expensive, leading to Complex Instruction Set Computers (CISC) like x86. CISC instructions are variable-length and can execute complex, multi-step operations (like moving data directly from one memory address to another) to keep code size small. As memory became cheaper and CPU speed outpaced RAM, Reduced Instruction Set Computers (RISC) emerged. RISC mandates fixed-length instructions and a load/store architecture—meaning the ALU only operates on registers, forcing memory access into dedicated instructions. This simplification allows for aggressive pipelining and higher clock speeds.

Chapter 4

CPU Control Unit Design

The brain within the brain.

While the datapath provides the physical wires and ALUs, it is completely inert without the Control Unit. The Control Unit acts as the orchestra conductor, decoding the opcode of the instruction and firing precisely timed voltage signals to dozens of multiplexers, register write-enables, and ALU control lines.

1. Hardwired Control

In a Hardwired Control Unit, the logic is implemented entirely with static combinational logic gates (AND/OR/Decoders) and state flip-flops. It operates incredibly fast because electricity simply flows through the silicon maze to generate the control signals. However, it is insanely difficult to design, impossible to patch if a bug is found after fabrication, and grows exponentially complex as the ISA expands. Most RISC processors utilize hardwired control for speed.

2. Microprogrammed Control

CISC architectures often use Microprogrammed Control to tame the complexity of their massive instruction sets. Inside the CPU is a tiny, hidden ROM (Read-Only Memory). Each complex assembly instruction acts as an address to this ROM. When fetched, it triggers a sequence of "microinstructions"—invisible, atomic hardware operations that execute over several clock cycles to complete the overarching complex instruction. This allows CPU designers to "write software" to control the hardware, making it easier to patch CPU bugs via microcode updates.

Chapter 5

Datapath Architecture

The physical journey of a bit.

The Datapath consists of all the components that actually hold and process data: the Register File, the ALU, multiplexers, and the data buses connecting them. Understanding the datapath is essential for understanding performance bottlenecks.

1. The Single-Cycle Datapath

In a pure single-cycle architecture, every instruction completes in one monumental clock cycle. The clock period must be stretched to accommodate the absolute slowest instruction in the ISA (typically a memory Load instruction, which must traverse instruction memory, register file read, ALU for address calculation, data memory read, and register file write). This is terribly inefficient because fast instructions (like a simple register ADD) are forced to sit idle, waiting for the massive clock cycle to finish.

2. The Multi-Cycle Datapath

A multi-cycle datapath breaks instruction execution into discrete, smaller steps (Fetch, Decode, Execute, Memory, Write-Back). By isolating components into these stages with intermediate buffer registers, the clock can run much faster. A simple ADD might finish in 3 fast clock cycles, while a complex LOAD takes 5. This drastically improves overall throughput.

Chapter 6

Pipelining & Instruction-Level Parallelism

The factory assembly line of computation.

Pipelining is the most profound architectural leap in CPU history. Rather than waiting for an instruction to completely finish before starting the next one, a pipelined datapath operates like a factory assembly line. While Instruction 1 is in the Execute stage, Instruction 2 is in the Decode stage, and Instruction 3 is being Fetched. Ideally, one instruction completes every single clock cycle.

1. Pipeline Hazards

This overlapping execution introduces terrifying complexities known as hazards.

  • Data Hazards: Occur when an instruction needs the result of a previous instruction that hasn't finished its write-back yet. Hardware solves this via Data Forwarding—intercepting the result directly from the ALU output and routing it back to the input before it hits the registers.
  • Control Hazards: Branch instructions (like `if` statements) ruin pipelining. The CPU doesn't know which instruction to fetch next until the branch is evaluated. Modern CPUs use Branch Prediction—guessing the outcome of an `if` statement based on historical patterns to keep the pipeline full. If it guesses wrong, it must frantically flush the pipeline.
  • Structural Hazards: Arise when two instructions in the pipeline need the same physical hardware resource simultaneously.

Chapter 7

Memory Hierarchy & Cache Design

Bridging the processor-memory gap.

Since the 1980s, CPU speeds have increased exponentially, while DRAM latency has barely improved. If a CPU had to fetch every variable directly from main memory, it would spend 99% of its life frozen, waiting for data. The solution is the Memory Hierarchy: placing small, blazing-fast SRAM (Cache) directly on the CPU die.

1. The Principle of Locality

Caches work because of human programming behavior. Temporal Locality means if you access a variable now, you will likely access it again very soon (like a loop counter). Spatial Locality means if you access an array at index 0, you will likely access index 1 next. When a CPU asks for a byte, the cache hardware automatically fetches a large "Block" or "Line" of 64 contiguous bytes, betting on spatial locality.

2. Cache Mapping and Replacements

Where does a block of data go when it enters the cache? Direct Mapped caches force each memory block into one exact slot, which is fast but leads to high conflict misses. Fully Associative caches let a block go anywhere, which is highly efficient but requires massively complex hardware to search. Set-Associative caches offer a middle ground. When the cache is full, replacement policies like LRU (Least Recently Used) dictate which old block gets evicted back to RAM to make room for new data.

Chapter 8

Virtual Memory

The grand illusion of infinite RAM.

Every process running on a modern OS believes it has complete, solitary control over gigabytes of contiguous RAM. This is an illusion orchestrated jointly by the Operating System and a physical hardware component called the MMU (Memory Management Unit).

1. Paging and Address Translation

Physical RAM is chopped into fixed-size chunks called Page Frames (usually 4KB). A program generates Virtual Addresses. The hardware uses a Page Table (stored in RAM) to translate the Virtual Page Number into a Physical Frame Number. If the OS ran out of physical RAM, it might have swapped that page to the hard drive. Accessing it triggers a hardware interrupt called a Page Fault, pausing the program until the disk retrieves it.

2. The TLB (Translation Lookaside Buffer)

Address translation poses a severe performance penalty. To read a variable, the CPU must access RAM once to read the Page Table, then access RAM again to get the actual data. It doubles the memory latency. To fix this, CPUs feature a specialized, ultra-fast cache called the Translation Lookaside Buffer (TLB). The TLB caches the most recent virtual-to-physical address mappings. A TLB hit allows translation in a fraction of a nanosecond.

Chapter 9

I/O Organization

Communicating with the outside world.

A CPU cannot just directly talk to a keyboard or a network card. It communicates through buses (like PCIe) and I/O Controllers. Handling asynchronous events from external devices without crippling CPU performance requires sophisticated hardware interventions.

1. Interrupt-Driven I/O

Polling (constantly checking if a device is ready) wastes billions of clock cycles. Instead, devices use hardware Interrupts. An interrupt is a literal electrical signal sent to a specific pin on the CPU. When asserted, the CPU stops whatever it is doing, saves its current state to the stack, and jumps to a specific memory address to run an Interrupt Service Routine (ISR) in the kernel to handle the incoming data.

2. Direct Memory Access (DMA)

If you are downloading a 10GB file from a 10-Gigabit network card, an interrupt for every byte would paralyze the CPU. The solution is a DMA Controller. The CPU simply tells the DMA controller: "Here is the network card port, here is the starting RAM address, move 10GB." The DMA controller takes over the system bus, moving the data straight from the network card into RAM autonomously, completely bypassing the CPU. The CPU is free to keep running your video game, and the DMA only interrupts the CPU once when the entire 10GB transfer is fully complete.

Chapter 10

Multiprocessors & Thread-Level Parallelism

Coordinating the multi-core chaos.

We hit the physical limitations of clock speed (power and heat dissipation walls) in the mid-2000s. To increase performance, the industry pivoted to Thread-Level Parallelism: stamping multiple independent CPU cores onto a single silicon die.

1. Symmetric Multiprocessing (SMP)

In SMP systems, multiple cores share the same physical RAM. This introduces the nightmare of Cache Coherence. If Core 0 reads variable X (value: 5) into its private L1 cache, and Core 1 reads variable X into its L1 cache, we have two copies. If Core 0 modifies X to 10, Core 1 now has stale, invalid data in its cache. A program running on Core 1 will crash or calculate incorrect results.

2. Cache Coherence Protocols (MESI)

Hardware solves this invisibly using "Snooping" protocols. The most common is MESI (Modified, Exclusive, Shared, Invalid). The caches constantly monitor the memory bus. When Core 0 writes to its cached copy of X, its cache controller broadcasts an "Invalidate" signal on the bus. Core 1 hears this, checks its own cache, realizes it holds X, and marks its copy as 'Invalid'. The next time Core 1 needs X, it suffers a cache miss and is forced to fetch the fresh '10' from Core 0 or Main Memory. Designing this state machine to run efficiently across dozens of cores without saturating the bus is one of the hardest problems in modern architecture.
